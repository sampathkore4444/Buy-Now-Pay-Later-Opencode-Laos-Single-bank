## 1. Executive Summary

This document specifies a **standalone, merchant-facing Buy Now, Pay Later (BNPL) platform** for the Lao PDR, modeled after the **Paidy architecture** (Japan). The platform operates **outside of any single bank's core banking system** (CBS), functioning as an independent credit and payments layer that connects to:
- **Multiple banks** (for customer funding, repayment collection, and settlement)
- **Merchants** (for checkout integration, transaction processing, and merchant settlement)
- **Customers** (for credit underwriting, BNPL line management, and repayment)

### Paidy-Style Architecture Principles
| Principle | Paidy (Japan) | Laos Adaptation |
|-----------|---------------|-----------------|
| **Standalone platform** | Not owned by one bank; bank-agnostic | Multi-bank connectivity via APIs |
| **Merchant-first** | Merchant enrollment drives consumer adoption | Lao QR + API + OTC integration |
| **Instant merchant settlement** | Paidy pays merchant immediately; takes repayment risk | Platform pays merchant T+1; carries credit risk |
| **Consumer signup once** | Use at any Paidy merchant | Single BNPL wallet, all merchants |
| **Real-time credit** | 3-second approval at checkout | < 2-second approval via pre-approved limit |
| **Revenue from MDR** | 3.5%–5% merchant fee | 3.5%–6.5% MDR (Laos risk-adjusted) |
| **No consumer interest** (if on time) | 0% for 30 days | 0% for 30 days; late fees only |

### Key Differentiator from Bank-Embedded BNPL
| Bank-Embedded BNPL | Standalone Paidy-Style Platform |
|--------------------|----------------------------------|
| Single bank only | Multi-bank (any customer from any partner bank) |
| Uses bank's CBS | Platform has its own ledger, risk engine, merchant system |
| Merchant is passive (just accepts QR) | Merchant actively enrolled, integrated, and settled by platform |
| Revenue: bank only | Revenue: platform shares MDR with partner banks |
| Limited to bank's customers | Open to any banked customer in Laos |

## 2. Laos Market Context & Platform Positioning

### 2.1 Why Standalone (Not Bank-Embedded)?
| Factor | Bank-Embedded Limitation | Standalone Platform Advantage |
|--------|--------------------------|------------------------------|
| **Customer reach** | Only Bank A's customers | All customers from Banks A, B, C, D |
| **Merchant appeal** | "Only for Bank A customers" | "Accept BNPL from any bank" |
| **Scale** | Limited by single bank's deposit base | Aggregated across banking sector |
| **Risk diversification** | Concentrated in one bank's portfolio | Pooling across customer bases |
| **Merchant settlement** | Bank pays itself (internal transfer) | Platform pays merchant from float |
| **Innovation speed** | Bank IT bureaucracy | Agile fintech stack |

### 2.3 Platform Economics (Laos-Calibrated)
| Parameter | Value | Rationale |
|-----------|-------|-----------|

## 4. Merchant Integration Models (Detailed)

### 4.1 Model A: Lao QR Overlay (Offline Retail)

**Target:** 70% of merchants — street shops, markets, restaurants, pharmacies

**How it works:**
1. Merchant already has a **Lao QR** from their bank (BCEL OnePay, LDB Pay, etc.)
2. Platform generates a **BNPL overlay QR** that wraps the merchant's existing QR
3. Customer scans with **LaoPay Flex app** (or any partner bank app with BNPL module)
4. App detects BNPL-enabled merchant → shows "Pay with LaoPay Flex" option
5. Customer confirms → platform pays merchant immediately → customer owes platform

**Technical Implementation:**
```
Merchant's Existing QR (Bank QR)
  |
  v
+----------------------------------+
|  BNPL Platform QR Overlay        |
|  +----------------------------+ |
|  |  Bank QR Payload           | |
|  |  + BNPL Merchant ID        | |
|  |  + BNPL Eligibility Flag   | |
|  +----------------------------+ |
+----------------------------------+
  |
  v
Customer scans with LaoPay Flex app
  |
  v
App decodes: "This is BNPL-enabled merchant M-123"
  -> Query merchant status
  -> Display BNPL option
  -> Process transaction
```

**Merchant Effort:** Zero. Platform sales team pastes a sticker over existing QR.

### 4.2 Model B: Dynamic QR (Semi-Integrated)

**Target:** 20% of merchants — shops with smartphones/tablets, small chains

**How it works:**
1. Merchant uses **LaoPay Flex Merchant App** (free download)
2. Enters amount → App generates **dynamic QR** with transaction-specific payload
3. Customer scans → amount pre-filled → instant approval
4. Merchant receives push notification: "Payment received: 450,000 LAK"

**Merchant App Features:**
- Generate dynamic QR per transaction
- View daily sales / settlement status
- Issue refunds (within 24 hours)
- Customer analytics (repeat rate, average ticket)
- Lao language interface

### 4.3 Model C: E-commerce API (Online)

**Target:** 5% of merchants — Lao e-commerce sites, Facebook sellers with websites

**Integration Flow:**

### 4.4 Model D: Pay-by-Link (Invoice / Social Commerce)

**Target:** 5% of merchants — Facebook sellers, clinics, schools, utilities

**How it works:**
1. Merchant generates payment link in Merchant App or Portal
2. Shares link via Facebook Messenger, WhatsApp, SMS, email
3. Customer clicks link → opens BNPL checkout page
4. Customer approves → merchant notified

**Use Case:**
- Facebook seller posts item → Customer messages "I want this" → Seller sends BNPL link
- Clinic sends invoice via SMS → Patient clicks → Pays via BNPL
- School sends tuition link → Parent pays via BNPL over 3 months

### 4.5 Model E: OTC Branch-Assisted (Rural)

**Target:** Rural merchants without smartphones

**How it works:**
1. Merchant enrolls via bank branch (partner bank acts as agent)
2. Branch staff creates merchant record in platform
3. Merchant receives printed **static QR** + "Accepts LaoPay Flex" sign
4. Customer scans at rural shop → uses any partner bank app
5. If customer has no smartphone, branch staff can process via **agent-assisted BNPL**

## 5. Revenue Model & MDR Structure

### 5.1 Merchant Discount Rate (MDR) Tiers

| Merchant Tier | MDR Rate | Criteria | Settlement |
|--------------|----------|----------|------------|
| **Tier 1: Strategic** | 3.5% | > 500M LAK monthly volume, national chain, low risk | T+1 |
| **Tier 2: Standard** | 4.5% | 50M–500M LAK monthly, established business, Green risk | T+1 |
| **Tier 3: Growth** | 5.5% | < 50M LAK monthly, new merchant, Yellow risk | T+1 |
| **Tier 4: High Risk** | 6.5% | Red risk category, high chargeback history, new sector | T+7 |

### 5.2 MDR Revenue Split

For every 100,000 LAK transaction at 4.5% MDR:

| Party | Share | Amount | Rationale |
|-------|-------|--------|-----------|
| **BNPL Platform** | 60% | 2,700 LAK | Credit risk, technology, merchant acquisition |
| **Partner Bank** | 40% | 1,800 LAK | Customer relationship, repayment collection, KYC |
| **Total MDR** | 100% | 4,500 LAK | |

### 5.3 Consumer-Side Revenue (Secondary)

| Fee Type | Amount | Trigger | Purpose |
|----------|--------|---------|---------|
| **Late fee** | 10,000 – 25,000 LAK | Missed payment after 3-day grace | Deterrent + cost recovery |
| **Post-30-day interest** | 1.5% monthly (BOL cap) | Balance unpaid after 30 days | Deterrent, not profit driver |
| **Reactivation fee** | 5,000 LAK | Account frozen due to default, then cleared | Administrative cost |

> **Philosophy:** Primary revenue is **MDR from merchants**. Consumer fees are for **risk management and cost recovery only**, not profit.

## 6. Settlement & Reconciliation Engine

### 6.1 Settlement Flow (T+1)

```
Day 0 (Transaction Day):
  10:00 AM — Customer buys 500,000 LAK TV via BNPL
  10:01 AM — Platform approves transaction
  10:02 AM — Merchant receives "Payment Confirmed" notification
  10:05 AM — Platform records: Merchant owes 500,000 LAK (gross)
                              Platform owes Merchant 477,500 LAK (net of 4.5% MDR)

Day 1 (T+1 Settlement):
  06:00 AM — Settlement batch runs
  06:30 AM — Platform transfers 477,500 LAK to Merchant's settlement account (Bank A)
  07:00 AM — Merchant receives SMS: "Settlement received: 477,500 LAK"
  07:00 AM — Platform's float account (Bank B) debited 477,500 LAK

Day 30 (Customer Repayment):
  06:00 AM — Platform auto-debits customer's account (Bank C) for 500,000 LAK
  06:30 AM — Platform receives 500,000 LAK into collection account
  06:30 AM — Platform's float replenished
```

### 6.2 Float Management

The platform maintains a **settlement float** to pay merchants T+1 **before** collecting from customers.

| Float Component | Amount | Purpose |
|-----------------|--------|---------|
| **Initial float** | 5,000,000,000 LAK | Seed capital from platform investors / bank consortium |
| **Daily replenishment** | Variable | Customer repayments auto-credited to float |
| **Buffer requirement** | 20% of daily volume | Safety margin for settlement failures |
| **Float bank** | Partner bank (BCEL or LDB) | Holding account for settlement funds |

### 6.3 Multi-Bank Reconciliation

Since customers and merchants can be at **different banks**:

```
+----------------------------------------------------------+
|              BNPL PLATFORM FLOAT ACCOUNT                 |
|              (Held at Partner Bank — BCEL)               |
+----------------------------------------------------------+
        |                              |
        | Pay merchants T+1            | Collect from customers
        v                              v
+---------------+              +---------------+
| Merchant at   |              | Customer at   |
| Bank B (LDB)  |              | Bank C (STB)  |
| Settlement:   |              | Repayment:    |
| 477,500 LAK   |              | 500,000 LAK   |
+---------------+              +---------------+
        |                              |
        |                              |
        v                              v
+----------------------------------------------------------+
|  INTERBANK SETTLEMENT (Lao Payment System / BOL)        |
|  - Net settlement across partner banks daily            |
|  - Platform reconciles: Float vs. Collections vs. Payouts |
+----------------------------------------------------------+
```

**Reconciliation API:**
- Daily batch reconciliation at 05:00 AM ICT
- Match: Transactions → Settlements → Collections → Exceptions
- Exception handling: Unmatched settlements, failed auto-debits, chargebacks

## 7. Merchant Risk Management

### 7.1 Merchant Risk Scoring

| Factor | Weight | Data Source |
|--------|--------|-------------|
| Business tenure | 15% | Registration date |
| Transaction volume trend | 20% | Platform history |
| Category risk | 15% | MCC mapping |
| Chargeback rate | 25% | Platform dispute data |
| Settlement history | 15% | On-time settlement of MDR |
| Customer complaint rate | 10% | Platform support tickets |

### 7.2 Merchant Risk Tiers & Actions

| Tier | Score | MDR | Settlement | Actions |
|------|-------|-----|------------|---------|
| **Green** | 80–100 | Standard | T+1 | Normal operations, volume bonuses |
| **Yellow** | 60–79 | +1% premium | T+1 | Enhanced monitoring, monthly review |
| **Orange** | 40–59 | +2% premium | T+3 | Weekly review, settlement hold possible |
| **Red** | < 40 | +3% premium or suspension | T+7 or hold | Immediate investigation, possible suspension |

### 7.3 Fraud Detection (Merchant Side)

| Fraud Type | Detection Method | Mitigation |
|------------|------------------|------------|
| **Fake transactions** | Merchant + customer collusion; same IP; round amounts | Velocity checks, device fingerprinting, manual review |
| **Refund abuse** | Excessive refunds vs. sales | Refund rate cap (>10% triggers review) |
| **Category misrepresentation** | Merchant claims "grocery" but sells electronics | Random audits, customer feedback analysis |
| **Transaction splitting** | Merchant splits large purchase to bypass limits | Link analysis, velocity checks |
| **Ghost merchant** | Merchant has no physical presence, no website | GPS verification, site visits for high-volume |

| `POST /accounts/{id}/credit` | Settlement to merchant | Credit merchant account T+1 |
| `GET /transactions/{id}` | Reconciliation | Match platform records with bank |
| `POST /webhooks/register` | Event notifications | Bank sends account events |

**Standardization:** Each bank implements a **BNPL adapter** that maps their internal APIs to the platform's standard interface.
