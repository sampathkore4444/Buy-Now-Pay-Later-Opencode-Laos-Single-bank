## 1. Executive Summary

This specification defines the end-to-end integration pattern for a single bank connection using **Option 1: Staging Table Posting with CBS End-of-Day (EOD) Pickup**. 

Under this model, the integration middleware writes financial and non-financial transactions into dedicated staging tables residing in the CBS database (or a logically segregated staging schema). The Core Banking System's nightly EOD batch identifies, validates, and posts these staged records into the live general ledger (GL) and sub-ledgers during its batch window. No real-time posting to live accounts occurs.

### 1.1 When to Use This Option
- CBS does not expose real-time posting APIs (or they are prohibitively expensive/unstable).
- Transaction volume is high but latency tolerance is 24 hours (e.g., retail sweep, interest accrual, fee batching).
- Regulatory or audit requirements mandate a "review gate" before ledger impact.
- The bank's technology roadmap has EOD batch as the canonical posting mechanism.
- Disaster recovery (DR) requirements favor idempotent batch replay over real-time reconciliation.

### 1.2 Key Principles
1. **Staging tables are the system of record until EOD confirms.**
2. **Never write directly to live GL tables.**
3. **Every staged record must be traceable to a unique integration correlation ID.**
4. **EOD pickup is pull-based; the integration layer must not push into live ledgers.**
5. **Idempotency is mandatory at every handoff point.**

### 2.1 Component Responsibilities

| Component | Responsibility | Owner |
|-----------|---------------|-------|
| **Integration Middleware** | Validate, enrich, deduplicate, and write to staging tables. Generate correlation IDs. | Integration Team |
| **Staging Schema** | Hold data in a transactional state until EOD. Provide ACID guarantees. | DBA / CBS Team |
| **EOD Batch Job** | Poll control tables, lock staging records, validate business rules, post to live GL, update status. | CBS Team |
| **Reconciliation Service** | Compare staging counts/sums against live ledger impacts. Trigger alerts on mismatch. | Integration Team |
| **Monitoring & Alerting** | Track staging table depth, EOD completion time, error rates. | DevOps / SRE |

## 3. Staging Table Schema (Normative)

All staging tables MUST reside in a dedicated schema (e.g., `INT_STG`) with restricted access. The integration user has `INSERT/SELECT` only. The CBS EOD user has `SELECT/UPDATE/DELETE` (soft delete).

### 3.1 STG_TXN_HEADER — Transaction Header

```sql
CREATE TABLE INT_STG.STG_TXN_HEADER (
    -- Primary Key & Correlation
    STG_HEADER_ID           BIGINT          GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    CORRELATION_ID          VARCHAR(64)     NOT NULL,  -- UUID v4 from middleware
    BATCH_ID                VARCHAR(64)     NOT NULL,  -- Logical batch grouping

    -- Source Identification
    SOURCE_SYSTEM           VARCHAR(32)     NOT NULL,  -- e.g., 'PAYMENT_HUB', 'LOANS_SYS'
    SOURCE_REF_NO           VARCHAR(128)    NOT NULL,  -- Transaction ID in source
    SOURCE_TIMESTAMP        TIMESTAMP(3)    NOT NULL,  -- When source created txn

    -- Transaction Classification
    TXN_TYPE                VARCHAR(16)     NOT NULL,  -- 'DEBIT', 'CREDIT', 'TRANSFER'
    TXN_CATEGORY            VARCHAR(32)     NOT NULL,  -- 'FEE', 'INTEREST', 'PRINCIPAL', 'ADJUSTMENT'
    TXN_CODE                VARCHAR(16)     NOT NULL,  -- CBS-specific transaction code

    -- Financial Summary
    TXN_CURRENCY            CHAR(3)         NOT NULL,  -- ISO 4217
    TXN_AMOUNT              DECIMAL(19,4)   NOT NULL,
    TXN_AMOUNT_LCY          DECIMAL(19,4)   NULL,       -- Local currency equivalent
    EXCHANGE_RATE           DECIMAL(19,8)   NULL,       -- Applied rate

    -- Accounts (Mandatory for financial txns)
    DEBIT_ACCOUNT_NO        VARCHAR(34)     NULL,       -- IBAN or internal account
    CREDIT_ACCOUNT_NO       VARCHAR(34)     NULL,
    DEBIT_ACCOUNT_BRANCH    VARCHAR(8)      NULL,
    CREDIT_ACCOUNT_BRANCH   VARCHAR(8)      NULL,

    -- Status Lifecycle
    STG_STATUS              VARCHAR(16)     NOT NULL DEFAULT 'PENDING',
                                -- PENDING, PICKED, POSTED, FAILED, REVERTED, HELD
    STG_STATUS_TIMESTAMP    TIMESTAMP(3)    NOT NULL DEFAULT CURRENT_TIMESTAMP,

    -- EOD Processing Metadata
    EOD_BATCH_RUN_ID        VARCHAR(64)     NULL,       -- Populated by EOD when picked
    EOD_PICKUP_TIMESTAMP    TIMESTAMP(3)    NULL,
    EOD_POSTING_TIMESTAMP   TIMESTAMP(3)    NULL,
    EOD_FAILURE_REASON      VARCHAR(512)    NULL,

    -- Retry & Control
    RETRY_COUNT             SMALLINT        NOT NULL DEFAULT 0,
    MAX_RETRIES             SMALLINT        NOT NULL DEFAULT 3,
    HOLD_REASON             VARCHAR(128)    NULL,       -- Why held (e.g., 'AML_CHECK')

    -- Audit
    CREATED_BY              VARCHAR(32)     NOT NULL DEFAULT 'INTEGRATION',
    CREATED_TIMESTAMP       TIMESTAMP(3)    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    MODIFIED_BY             VARCHAR(32)     NULL,
    MODIFIED_TIMESTAMP      TIMESTAMP(3)    NULL,

    -- Constraints
    CONSTRAINT uq_correlation UNIQUE (CORRELATION_ID),
    CONSTRAINT chk_status CHECK (STG_STATUS IN ('PENDING','PICKED','POSTED','FAILED','REVERTED','HELD')),
    CONSTRAINT chk_amount_positive CHECK (TXN_AMOUNT >= 0),
    CONSTRAINT chk_debit_or_credit CHECK (
        (TXN_TYPE = 'DEBIT' AND DEBIT_ACCOUNT_NO IS NOT NULL) OR
        (TXN_TYPE = 'CREDIT' AND CREDIT_ACCOUNT_NO IS NOT NULL) OR
        (TXN_TYPE = 'TRANSFER' AND DEBIT_ACCOUNT_NO IS NOT NULL AND CREDIT_ACCOUNT_NO IS NOT NULL)
    )
);

-- Indexes for EOD performance
CREATE INDEX idx_stg_header_status ON INT_STG.STG_TXN_HEADER(STG_STATUS) WHERE STG_STATUS = 'PENDING';
CREATE INDEX idx_stg_header_batch ON INT_STG.STG_TXN_HEADER(BATCH_ID, STG_STATUS);
CREATE INDEX idx_stg_header_source ON INT_STG.STG_TXN_HEADER(SOURCE_SYSTEM, SOURCE_REF_NO);
CREATE INDEX idx_stg_header_eod_run ON INT_STG.STG_TXN_HEADER(EOD_BATCH_RUN_ID);
CREATE INDEX idx_stg_header_created ON INT_STG.STG_TXN_HEADER(CREATED_TIMESTAMP);
```

### 3.3 STG_TXN_CONTROL — Batch Control & Handshake

```sql
CREATE TABLE INT_STG.STG_TXN_CONTROL (
    BATCH_ID                VARCHAR(64)     PRIMARY KEY,
    SOURCE_SYSTEM           VARCHAR(32)     NOT NULL,

    -- Batch Metrics (Populated by Integration BEFORE posting records)
    EXPECTED_RECORD_COUNT   INTEGER         NOT NULL,
    EXPECTED_TOTAL_AMOUNT   DECIMAL(19,4)   NOT NULL,
    EXPECTED_TOTAL_AMOUNT_LCY DECIMAL(19,4) NULL,

    -- Actuals (Populated by EOD AFTER processing)
    ACTUAL_RECORD_COUNT     INTEGER         NULL,
    ACTUAL_TOTAL_AMOUNT     DECIMAL(19,4)   NULL,
    ACTUAL_TOTAL_AMOUNT_LCY DECIMAL(19,4)   NULL,

    CONTROL_STATUS          VARCHAR(16)     NOT NULL DEFAULT 'OPEN',

    -- EOD Metadata
    EOD_BATCH_RUN_ID        VARCHAR(64)     NULL,
    EOD_START_TIMESTAMP     TIMESTAMP(3)    NULL,
    EOD_END_TIMESTAMP       TIMESTAMP(3)    NULL,

    -- Reconciliation Flags
    RECONCILE_STATUS        VARCHAR(16)     NULL,       -- MATCHED, MISMATCH, PENDING
    RECONCILE_DIFFERENCE    DECIMAL(19,4)   NULL,
    RECONCILE_TIMESTAMP     TIMESTAMP(3)    NULL,

    CREATED_TIMESTAMP       TIMESTAMP(3)    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    MODIFIED_TIMESTAMP      TIMESTAMP(3)    NULL,

    CONSTRAINT chk_control_status CHECK (CONTROL_STATUS IN ('OPEN','READY_FOR_PICKUP','IN_PROGRESS','COMPLETED','PARTIAL','FAILED'))
);
```

## 4. Data Flow & State Machine

### 4.1 Integration Middleware Flow (Write Side)

```
┌─────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ Receive │───▶│ Validate    │───▶│ Enrich      │───▶│ Deduplicate │───▶│ Write to    │
│ Source  │    │ Schema &    │    │ (Rates,     │    │ (Correlation│    │ Staging     │
│ Payload │    │ Business    │    │ GL Codes)   │    │ ID Check)   │    │ Tables      │
│         │    │ Rules       │    │             │    │             │    │             │
└─────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
                                                                               │
                                                                               ▼
                                                                        ┌─────────────┐
                                                                        │ Update      │
                                                                        │ CONTROL     │
                                                                        │ Table       │
                                                                        └─────────────┘
```

#### Step-by-Step:

**Step 1: Receive & Parse**
- Accept payload via secured API (REST/gRPC/MQ) or file drop.
- Validate JSON/XML schema against published XSD/JSON Schema.
- Reject immediately if schema invalid; return `400` with error code `SCH-001`.

**Step 2: Validate Business Rules**
- Account existence check (query read-only account master view).
- Currency code against bank's approved currency list.
- Amount precision (max 4 decimal places).
- Transaction code validity against CBS txn code master.
- Date validation: `SOURCE_TIMESTAMP` must not be future-dated by >24h or back-dated by >90 days (configurable).

**Step 3: Enrich**
- Fetch exchange rate if `TXN_CURRENCY != LCY`.
- Map `TXN_CODE` to GL accounts via integration mapping table.
- Populate `DEBIT_ACCOUNT_BRANCH` / `CREDIT_ACCOUNT_BRANCH` from account master.
- Generate `CORRELATION_ID` = `UUIDv4`.
- Assign `BATCH_ID` based on time window (e.g., `YYYYMMDD_HH` for hourly batches).

**Step 4: Deduplicate**
- Query `STG_TXN_HEADER` for existing `SOURCE_SYSTEM + SOURCE_REF_NO`.
- If found with status `POSTED` or `PICKED`: return `409 Conflict` with existing `CORRELATION_ID`.
- If found with status `PENDING` or `HELD`: update if idempotent retry; else return `409`.
- If found with status `FAILED` and `RETRY_COUNT < MAX_RETRIES`: allow re-insert with new `STG_HEADER_ID` but same `CORRELATION_ID`? **NO** — see Section 6.2 (Retry Policy).

**Step 5: Write to Staging**
- Open database transaction.
- Insert into `STG_TXN_HEADER`.
- Insert 1..N rows into `STG_TXN_DETAIL`.
- If control record for `BATCH_ID` does not exist: insert into `STG_TXN_CONTROL` with `STATUS = 'OPEN'`.
- Commit transaction.
- If any insert fails: rollback entire transaction, log to `STG_ERROR_LOG`, return `500`.

**Step 6: Update Control Table**
- After all records in logical batch are written, update `STG_TXN_CONTROL`:
  - `STATUS = 'READY_FOR_PICKUP'`
  - `EXPECTED_RECORD_COUNT = COUNT(*)` from header for this batch
  - `EXPECTED_TOTAL_AMOUNT = SUM(TXN_AMOUNT)`
- This signals to EOD that the batch is complete and consistent.

### 4.2 EOD Batch Flow (Read/Process Side)

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ EOD Job     │───▶│ Lock & Pick │───▶│ Validate    │───▶│ Post to     │───▶│ Update      │
│ Starts      │    │ PENDING     │    │ Business    │    │ Live GL     │    │ Staging     │
│ (Scheduled) │    │ Batches     │    │ Rules       │    │             │    │ Status      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                                               │
                                                                               ▼
                                                                        ┌─────────────┐
                                                                        │ Reconcile   │
                                                                        │ & Report    │
                                                                        └─────────────┘
```

#### Step-by-Step:

**Step 1: EOD Initiation**
- Triggered by CBS scheduler at configurable time (default: 22:00 local time).
- Generate `EOD_BATCH_RUN_ID` = `EOD_YYYYMMDD_HHMMSS_<RANDOM>`.
- Log start in CBS batch log.

**Step 2: Identify Ready Batches**
```sql
SELECT BATCH_ID 
FROM INT_STG.STG_TXN_CONTROL 
WHERE CONTROL_STATUS = 'READY_FOR_PICKUP'
  AND CREATED_TIMESTAMP < CURRENT_TIMESTAMP - INTERVAL '15' MINUTE;  -- Grace period
```
- The 15-minute grace period prevents picking a batch that is still being written by the integration layer.

**Step 3: Lock & Pick Records**
- For each ready batch:
  - Update `STG_TXN_CONTROL` to `CONTROL_STATUS = 'IN_PROGRESS'`, `EOD_BATCH_RUN_ID = <id>`, `EOD_START_TIMESTAMP = NOW()`.
  - Update `STG_TXN_HEADER` for all `PENDING` records in batch:
    - `STG_STATUS = 'PICKED'`
    - `EOD_BATCH_RUN_ID = <id>`
    - `EOD_PICKUP_TIMESTAMP = NOW()`
  - Use `SELECT FOR UPDATE` or equivalent pessimistic locking to prevent concurrent EOD instances from picking same records.

**Step 4: Validate Business Rules (Second Pass)**
- Re-validate accounts (account may have been closed since staging).
- Check available balance for debit transactions (if balance-sensitive).
- Verify no duplicate `CORRELATION_ID` exists in live `TXN_HISTORY`.
- Check holiday calendar (prevent posting on bank holidays if configured).
- Validate FX rates are still within tolerance (if rate changed significantly).

**Step 5: Post to Live GL**
- For each validated record:
  - Insert into live `TXN_HISTORY` / `GL_ENTRIES` via CBS internal APIs (not direct table write).
  - Update account balances via CBS balance update routine.
  - If any posting fails, mark individual record as `FAILED` and continue with next (fail-one-continue-others mode).
  - Alternatively, if configured for `ALL_OR_NONE`, fail entire batch and mark all as `FAILED`.

**Step 6: Update Staging Status**
- Successful: `STG_STATUS = 'POSTED'`, `EOD_POSTING_TIMESTAMP = NOW()`.
- Failed: `STG_STATUS = 'FAILED'`, `EOD_FAILURE_REASON = <error_detail>`.
- Update `STG_TXN_CONTROL`:
  - `ACTUAL_RECORD_COUNT`, `ACTUAL_TOTAL_AMOUNT`
  - `CONTROL_STATUS = 'COMPLETED'` (if all posted) or `'PARTIAL'` (if some failed).
  - `EOD_END_TIMESTAMP = NOW()`.

**Step 7: Reconciliation & Reporting**
- Compare `EXPECTED_*` vs `ACTUAL_*` in control table.
- Insert into `STG_RECONCILE`.
- If mismatch: set `RECONCILE_STATUS = 'MISMATCH'`, trigger alert.
- Generate EOD report (CSV/PDF) for operations team.

## 6. Edge Cases & Exception Handling

### 6.1 Duplicate Submission (Same Source Ref, Same Payload)

**Scenario:** Integration retries due to network timeout; source system sends same txn twice.

**Detection:**
- Primary key on `CORRELATION_ID` (generated by integration) prevents exact duplicate.
- Unique constraint on `(SOURCE_SYSTEM, SOURCE_REF_NO)` prevents logical duplicate.

**Handling:**
1. If `CORRELATION_ID` matches: return `200 OK` with current status (idempotent read).
2. If `SOURCE_SYSTEM + SOURCE_REF_NO` matches but `CORRELATION_ID` differs: return `409 Conflict` with existing record details.
3. If source system claims this is a legitimate correction (not duplicate): source must send a **reversal** first, then new transaction.

### 6.2 Retry Policy for Failed Records

**Scenario:** EOD marks record as `FAILED` due to temporary issue (e.g., account locked).

**Policy:**
- `MAX_RETRIES = 3` (configurable per `TXN_CATEGORY`).
- When EOD fails a record, increment `RETRY_COUNT`.
- If `RETRY_COUNT < MAX_RETRIES`:
  - Reset `STG_STATUS = 'PENDING'` (requires manual ops approval or automated retry job).
  - EOD next day will re-pick it.
- If `RETRY_COUNT >= MAX_RETRIES`:
  - Keep `STG_STATUS = 'FAILED'`.
  - Move to manual exception queue.
  - Alert operations team via email/SMS/Slack.

**Retry Window:**
- Retries only occur on subsequent EOD runs. Same EOD run does NOT retry failed records.
- Records held for retry must be excluded from `RECONCILE` until final disposition.

### 6.3 Partial Batch Failure

**Scenario:** 1000 records in batch; 998 post successfully, 2 fail.

**Handling:**
- Default mode: `FAIL_ONE_CONTINUE` (individual record failure does not block others).
- Alternative mode: `ALL_OR_NONE` (configured per `BATCH_ID` or `TXN_CATEGORY`).
- In `FAIL_ONE_CONTINUE`:
  - `STG_TXN_CONTROL.CONTROL_STATUS = 'PARTIAL'`.
  - `ACTUAL_RECORD_COUNT = 998`.
  - Failed records remain in `STG_STATUS = 'FAILED'`.
  - Reconciliation shows `DIFFERENCE_COUNT = 2`.
  - Operations must resolve failed records before batch can be marked `COMPLETED`.

### 6.4 EOD Batch Run Failure (Crashes Mid-Run)

**Scenario:** CBS EOD job crashes after picking 500 records but before completing.

**Detection:**
- Heartbeat table or process monitor detects EOD job death.
- `STG_TXN_CONTROL.EOD_END_TIMESTAMP` is NULL > threshold (e.g., 4 hours after `EOD_START_TIMESTAMP`).

**Recovery:**
1. **Do NOT auto-restart.** Operations must investigate.
2. Identify records with `STG_STATUS = 'PICKED'` and `EOD_BATCH_RUN_ID = <crashed_run>`.
3. Check live GL: if some records were actually posted before crash (check `TXN_HISTORY` for `CORRELATION_ID`), mark those as `POSTED`.
4. For remaining `PICKED` records: reset to `PENDING` (requires manual approval or automated rollback script).
5. Reset `STG_TXN_CONTROL` to `READY_FOR_PICKUP`.
6. Re-run EOD for affected batches only.

**Idempotency Guarantee:**
- Live GL insertions use `CORRELATION_ID` as unique key. Re-posting same `CORRELATION_ID` is rejected by CBS, preventing double-posting.

### 6.5 Staging Table Full / Performance Degradation

**Scenario:** Staging table grows to millions of rows; EOD pickup slows down.

**Mitigation:**
- Partition `STG_TXN_HEADER` by `CREATED_TIMESTAMP` (monthly partitions).
- Archive `POSTED` records > 90 days to `STG_TXN_HEADER_ARCHIVE`.
- Index maintenance: rebuild `idx_stg_header_status` nightly after EOD.
- Alert when table size > 10M rows or oldest `PENDING` record > 48 hours.

### 6.6 Back-Dated Transactions

**Scenario:** Source system sends transaction with `SOURCE_TIMESTAMP` from 30 days ago.

**Policy:**
- Configurable `MAX_BACKDATE_DAYS` (default: 90).
- If back-dated > threshold: reject at integration layer with `ERR-BACKDATE-001`.
- If within threshold: allow staging, but EOD applies back-date posting logic (if CBS supports it).
- If CBS does not support back-dating: EOD marks as `FAILED` with reason `BACKDATE_NOT_SUPPORTED`.

### 6.7 Account Closure Between Staging and EOD

**Scenario:** Account is staged as debit account at 14:00. Customer closes account at 16:00. EOD runs at 22:00.

**Handling:**
- EOD validation step re-checks account status.
- If account closed: mark as `FAILED`, reason `ACCOUNT_CLOSED`.
- If account dormant (no transactions > 1 year): may require supervisor override to post; default to `HELD` status pending ops review.

### 6.8 FX Rate Change Between Staging and EOD

**Scenario:** FX rate changes significantly between staging and EOD.

**Handling:**
- Store `EXCHANGE_RATE` at staging time.
- EOD re-validates rate against current rate.
- If deviation > `FX_TOLERANCE_PERCENT` (default 2%): mark as `HELD` for treasury review.
- If within tolerance: post using staged rate (rate is locked at staging time for accuracy).

### 6.9 Zero-Amount Transactions

**Scenario:** Source sends `TXN_AMOUNT = 0.0000`.

**Handling:**
- Integration layer rejects with `ERR-ZERO-001` unless `TXN_CATEGORY` explicitly allows zero-amount (e.g., `NOTIFICATION` category).
- Constraint `chk_amount_positive` allows zero; business rule layer enforces rejection.

### 6.10 Missing Mandatory Fields

**Scenario:** `DEBIT_ACCOUNT_NO` is null for a `TRANSFER` transaction.

**Handling:**
- Integration layer schema validation catches this before DB write.
- Returns `400 Bad Request` with field-level error details.
- Never reaches staging table.

### 6.11 CBS EOD Delayed / Missed

**Scenario:** CBS EOD does not run due to maintenance window or holiday.

**Handling:**
- Staging records remain in `PENDING` status.
- Integration monitoring alerts if oldest `PENDING` record > 36 hours.
- Operations can trigger **ad-hoc EOD** for specific batches if CBS supports it.
- If EOD skipped entirely (e.g., long weekend), next EOD run processes all accumulated batches.
- Ensure EOD batch job can handle > 72 hours of accumulated volume (performance tested).

### 6.12 Network Partition Between Integration and CBS DB

**Scenario:** Integration cannot reach CBS database during write.

**Handling:**
- Integration middleware queues transactions in local persistent store (e.g., local DB, Kafka).
- Retry with exponential backoff: 1s, 2s, 4s, 8s, 16s, 32s, then max 60s intervals.
- After 10 minutes of failure: circuit breaker opens; integration returns `503 Service Unavailable` to source.
- Source systems must implement their own retry or hold queue.
- Once connectivity restored: drain local queue to staging tables.

### 6.13 Concurrent EOD Instances (Race Condition)

**Scenario:** Two EOD instances accidentally start simultaneously (e.g., active-active DR setup).

**Handling:**
- Implement distributed lock using database advisory locks (`pg_advisory_lock` for PostgreSQL, `DBMS_LOCK` for Oracle, `sp_getapplock` for SQL Server).
- Lock key derived from constant (e.g., `12345` for EOD job).
- Second instance waits 5 minutes; if lock not acquired, exits with error.
- Alternatively, use `SELECT FOR UPDATE SKIP LOCKED` on `STG_TXN_CONTROL` to partition batches between workers (if parallel EOD is desired).

### 6.14 Data Corruption in Staging Tables

**Scenario:** DBA accidentally truncates staging table or corrupts data.

**Handling:**
- Staging schema must have **Point-in-Time Recovery (PITR)** enabled.
- Restore from backup to a separate recovery instance.
- Re-insert missing records from integration middleware's audit log (integration must retain source payloads for 30 days minimum).
- Cross-reference with source system to identify missing transactions.
- If records were posted to live GL before corruption: use `CORRELATION_ID` in `TXN_HISTORY` to verify.

### 6.15 Reversal of Already-Posted Transaction

**Scenario:** Source requests reversal of a transaction that was already picked up and posted by EOD.

**Handling:**
- Integration cannot simply delete or update staging record (it's already `POSTED`).
- Instead, create a **new staging record** with:
  - `TXN_TYPE = 'REVERSAL'`
  - `ORIGINAL_CORRELATION_ID = <original>`
  - Negative amount (if CBS supports) or offsetting debit/credit swap.
- EOD processes reversal as a normal transaction in next run.
- Link reversal to original in `STG_TXN_DETAIL.FLEX_FIELD_1`.

### 6.16 Large Batch Size (Millions of Records)

**Scenario:** End-of-month interest capitalization creates 5 million records.

**Handling:**
- Break into sub-batches using `BATCH_ID` suffix (e.g., `20260731_INT_001` through `20260731_INT_050`).
- Each sub-batch max 100,000 records.
- EOD processes sub-batches sequentially or in parallel (if parallel, ensure no inter-batch dependencies).
- Integration writes sub-batches with 5-minute gaps to avoid overwhelming the DB.
- Monitor DB connection pool and temp table space.

## 7. Security Specification

### 7.1 Database Access Control

| Role | Permissions | Scope |
|------|-------------|-------|
| `INT_STG_WRITER` | `INSERT` on `STG_TXN_HEADER`, `STG_TXN_DETAIL`; `SELECT` on `STG_TXN_CONTROL` | Integration middleware only |
| `INT_STG_READER` | `SELECT` on all staging tables | Reconciliation service, reporting |
| `CBS_EOD_USER` | `SELECT`, `UPDATE` on all staging tables; `INSERT` on `STG_ERROR_LOG`, `STG_RECONCILE` | EOD batch job only |
| `CBS_DBA` | Full DDL | Schema changes, partitioning |

- No direct `DELETE` permissions. Archival is handled via stored procedure.
- All access via service accounts; no personal accounts.
- Password rotation every 90 days; integration uses vault (HashiCorp Vault / AWS Secrets Manager).

### 7.2 Data Encryption

- **In Transit:** TLS 1.3 for all API calls; JDBC connections use SSL/TLS with certificate pinning.
- **At Rest:** Database TDE (Transparent Data Encryption) for staging schema.
- **Sensitive Fields:** Account numbers must be tokenized or masked in non-production environments. In production, access to plain account numbers requires dual control approval.

### 7.3 Audit Logging

- All `INSERT` operations to staging tables logged to `STG_AUDIT_TRAIL` (separate table or database trigger).
- Log fields: `OPERATION`, `TABLE_NAME`, `RECORD_ID`, `USER_ID`, `TIMESTAMP`, `CLIENT_IP`, `SESSION_ID`.
- Audit logs retained for 7 years per regulatory requirement.
- Tamper-evident: audit log table has write-once, append-only constraints.

### 7.4 API Security

- OAuth 2.0 / mTLS for service-to-service authentication.
- Rate limiting: 10,000 requests/minute per source system.
- IP whitelist: integration middleware IPs only.
- Payload size limit: 10 MB per request.
- Input sanitization: prevent SQL injection, XSS, and XML external entity (XXE) attacks.

## 9. Reconciliation Procedures

### 9.1 Daily Reconciliation (Automated)

**Triggered:** After EOD completion, before 06:00 next business day.

**Steps:**
1. For each `BATCH_ID` with `CONTROL_STATUS = 'COMPLETED'` or `'PARTIAL'`:
   - Sum `STG_TXN_HEADER.TXN_AMOUNT` where `STG_STATUS = 'POSTED'`.
   - Sum live `TXN_HISTORY.TXN_AMOUNT` where `SOURCE_REF_NO` matches.
   - Compare counts and amounts.
2. Insert results into `STG_RECONCILE`.
3. If `DIFFERENCE_AMOUNT != 0` or `DIFFERENCE_COUNT != 0`:
   - Set `RECONCILE_STATUS = 'UNBALANCED'`.
   - Trigger Critical alert.
   - Generate exception report for operations.

### 9.2 Manual Reconciliation (Break-Glass)

**When:** Automated reconciliation fails or is unavailable.

**Steps:**
1. Export `STG_TXN_HEADER` for date range where `STG_STATUS = 'POSTED'`.
2. Export `TXN_HISTORY` for same date range, filtered by `SOURCE_SYSTEM`.
3. Match on `CORRELATION_ID` (or `SOURCE_REF_NO` if correlation not propagated).
4. Identify orphans in either direction.
5. Investigate root cause (missing staging write, double EOD run, etc.).
6. Corrective action: reverse erroneous postings, re-stage missing records, or adjust control totals.

### 9.3 Month-End Reconciliation

- Additional validation: ensure all `PENDING` records from prior month are resolved.
- Archive `POSTED` records > 90 days to cold storage.
- Generate compliance attestation report.

## 10. Disaster Recovery & Business Continuity

### 10.1 Recovery Point Objective (RPO)
- **Staging Data:** 0 minutes (synchronous replication to DR site).
- **Integration Middleware Queue:** 5 minutes (asynchronous replication acceptable).

### 10.2 Recovery Time Objective (RTO)
- **Staging Table Access:** 1 hour (failover to DR database).
- **EOD Batch Processing:** 4 hours (can be deferred to next business day if intra-day).

### 10.3 DR Scenarios

| Scenario | Impact | Action |
|----------|--------|--------|
| Primary DB failure | Cannot write to staging | Failover to DR DB; integration middleware reconnects via connection string update. |
| Integration middleware failure | Cannot accept source payloads | Source systems queue locally; restore middleware from backup or activate hot standby. |
| CBS EOD failure | Posting delayed | Hold staging records; run EOD on DR CBS if available; else defer to next business day. |
| Complete site failure | All systems down | Activate DR site; replay integration queue from last checkpoint; reconcile upon restoration. |

### 10.4 Backup Strategy

- **Staging Database:** Full backup daily at 02:00; incremental every 4 hours; transaction log backup every 15 minutes.
- **Retention:** 35 days local, 1 year offsite.
- **Test Restore:** Monthly DR drill; validate backup integrity.

## 13. Operational Runbooks

### 13.1 Runbook: EOD Job Stuck

1. Check CBS batch monitor for EOD process status.
2. If running > 6 hours: check DB for blocking locks.
3. If blocked by long-running query: identify session; kill if safe.
4. If EOD process dead: follow Section 6.4 (EOD Crash Recovery).
5. Notify CBS team and integration architect.

### 13.2 Runbook: High Failed Record Count

1. Query `STG_ERROR_LOG` for `ERROR_PHASE = 'EOD_POSTING'` in last 24h.
2. Group by `ERROR_CODE` to identify pattern.
3. If `ACCOUNT_CLOSED`: check for bulk account closure event; notify relationship managers.
4. If `INSUFFICIENT_BALANCE`: check if source system sent pre-authorization debits that were not reserved.
5. If `FX_RATE_TOLERANCE`: notify treasury to review rate tolerance settings.
6. For systemic issues: halt source system ingestion until root cause resolved.

### 13.3 Runbook: Reconciliation Mismatch

1. Identify `BATCH_ID` with `RECONCILE_STATUS = 'UNBALANCED'`.
2. Compare `STG_RECONCILE` vs `STG_TXN_CONTROL` totals.
3. If control totals match but live GL differs: investigate CBS double-posting or missing reversal.
4. If staging totals differ from control: investigate integration middleware duplicate write or missed write.
5. Generate detailed diff report.
6. Escalate to CBS DBA and integration lead.

### 13.4 Runbook: Security Incident (Unauthorized Staging Access)

1. Immediately revoke compromised service account.
2. Query `STG_AUDIT_TRAIL` for all operations by compromised account in last 24h.
3. Identify any unauthorized inserts/updates.
4. If unauthorized records exist: quarantine by setting `STG_STATUS = 'HELD'`.
5. Notify InfoSec, Compliance, and Integration teams.
6. Force password rotation for all staging schema accounts.
7. Conduct forensic analysis; file incident report.

## 16. Appendices

### Appendix A: Sample EOD Stored Procedure (Pseudocode)

```sql
PROCEDURE EOD_PICKUP_AND_POST(
    IN p_eod_batch_run_id VARCHAR(64)
)
BEGIN
    DECLARE v_batch_id VARCHAR(64);
    DECLARE v_cursor CURSOR FOR
        SELECT BATCH_ID 
        FROM INT_STG.STG_TXN_CONTROL 
        WHERE CONTROL_STATUS = 'READY_FOR_PICKUP'
        FOR UPDATE SKIP LOCKED;

    OPEN v_cursor;
    FETCH_LOOP: LOOP
        FETCH v_cursor INTO v_batch_id;
        IF done THEN LEAVE FETCH_LOOP; END IF;

        -- Lock control
        UPDATE INT_STG.STG_TXN_CONTROL 
        SET CONTROL_STATUS = 'IN_PROGRESS',
            EOD_BATCH_RUN_ID = p_eod_batch_run_id,
            EOD_START_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id;

        -- Pick records
        UPDATE INT_STG.STG_TXN_HEADER
        SET STG_STATUS = 'PICKED',
            EOD_BATCH_RUN_ID = p_eod_batch_run_id,
            EOD_PICKUP_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id 
          AND STG_STATUS = 'PENDING';

        -- Process each record
        FOR EACH RECORD IN (
            SELECT * FROM INT_STG.STG_TXN_HEADER 
            WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'PICKED'
        ) LOOP
            BEGIN
                -- Validate
                IF NOT validate_account(DEBIT_ACCOUNT_NO) THEN
                    RAISE validation_error;
                END IF;

                -- Post to live GL via CBS API
                CBS_POST_TRANSACTION(RECORD);

                -- Mark posted
                UPDATE INT_STG.STG_TXN_HEADER
                SET STG_STATUS = 'POSTED',
                    EOD_POSTING_TIMESTAMP = NOW()
                WHERE STG_HEADER_ID = RECORD.STG_HEADER_ID;

            EXCEPTION WHEN OTHERS THEN
                UPDATE INT_STG.STG_TXN_HEADER
                SET STG_STATUS = 'FAILED',
                    EOD_FAILURE_REASON = SQLERRM,
                    RETRY_COUNT = RETRY_COUNT + 1
                WHERE STG_HEADER_ID = RECORD.STG_HEADER_ID;

                INSERT INTO INT_STG.STG_ERROR_LOG (...);
            END;
        END LOOP;

        -- Finalize control
        UPDATE INT_STG.STG_TXN_CONTROL
        SET CONTROL_STATUS = CASE 
                WHEN EXISTS (SELECT 1 FROM INT_STG.STG_TXN_HEADER 
                             WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'FAILED')
                THEN 'PARTIAL' ELSE 'COMPLETED' END,
            ACTUAL_RECORD_COUNT = (SELECT COUNT(*) FROM INT_STG.STG_TXN_HEADER WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'POSTED'),
            ACTUAL_TOTAL_AMOUNT = (SELECT SUM(TXN_AMOUNT) FROM INT_STG.STG_TXN_HEADER WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'POSTED'),
            EOD_END_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id;

    END LOOP;
    CLOSE v_cursor;
END;
```
