## 1. Executive Summary

This document specifies a **single bank-embedded Buy Now, Pay Later (BNPL) product** for the Lao PDR, designed to operate as a native product module within an existing commercial bank's technology estate. Unlike a standalone, multi-bank platform, this architecture leverages the bank's **Core Banking System (CBS)**, **general ledger**, **customer base**, and **regulatory license** to deliver BNPL services with lower capital requirements, faster time-to-market, and immediate customer trust.

### Key Architectural Decision: Hybrid Processing Model
| Layer | Timing | Mechanism | Purpose |
|-------|--------|-----------|---------|
| **Authorization** | Real-time (< 2 sec) | API + Cache (Redis) | Credit limit check, merchant validation, fraud screening |
| **Financial Posting** | Batch (T+1, EOD) | Staging Tables → CBS EOD | General ledger impact, merchant settlement, customer repayment |

### Why This Matters for Laos
- **No external float required**: The bank's balance sheet absorbs the T+1 settlement gap between merchant payout and customer collection.
- **Regulatory simplicity**: Operates under the bank's existing BOL license; no separate PSP or microfinance license required.
- **Instant trust**: Customers and merchants already have relationships with the bank.
- **Existing infrastructure**: KYC, AML, credit bureau connections, and merchant acquiring are already in place.

### Product Positioning
- **Consumer-facing**: "Pay in 30 days, 0% interest" for the bank's existing and new customers.
- **Merchant-facing**: T+1 settlement, competitive MDR (3.5%–6.5%), Lao QR integration.
- **Scope**: Phase 1 serves the bank's own customers and any merchant (same-bank or interbank via existing payment rails).

## 2. Strategic Rationale: Single Bank Embedded vs. Standalone

### 2.1 Laos Market Context
| Factor | Standalone Platform | Single Bank Embedded |
|--------|---------------------|----------------------|
| **Time to market** | 12–18 months (new licenses, float capital, bank negotiations) | 6–9 months (leverage existing license) |
| **Regulatory capital** | 5B–10B LAK float + separate licensing | Zero external float; uses bank balance sheet |
| **Customer acquisition** | Cold start; requires massive marketing | Immediate access to bank's depositors |
| **KYC / AML** | Build from scratch | Reuse bank's existing KYC/CDD infrastructure |
| **Credit risk data** | Limited bureau access; thin files | Rich internal data (salary, deposits, history) |
| **Merchant trust** | Unknown fintech brand | Trusted bank brand |
| **Technology cost** | Full stack build | BNPL layer only; reuse CBS, cards, payments |
| **Interbank reach** | Must negotiate with every bank | Bank's existing participation in Lao QR |
| **Revenue ownership** | Shared with partner banks | Bank retains 100% of MDR |

### 2.2 Addressing the Multi-Bank Limitation
The primary constraint of a single bank model is that **BNPL credit is only available to the bank's customers**. However:
1. **Merchant acceptance is universal**: Any merchant (regardless of their bank) can accept BNPL payments via Lao QR or API, because settlement is executed through the bank's existing payment rails.
2. **Future extensibility**: The architecture is designed to allow future "partner bank" connectivity via the same staging table interface, should the bank wish to white-label or consortium-expand in Phase 2.

### 3.2 Dual-Path Processing Explained

**Path A: Real-Time Authorization (Checkout Experience)**
1. Customer selects "Pay with BNPL" at merchant checkout.
2. BNPL Auth Engine queries Redis for pre-approved limit (< 50ms).
3. Merchant status and fraud rules checked in BNPL product database.
4. **Decision returned in < 2 seconds**. No live GL account is touched.
5. Auth record created with status `AUTHED`. Limit is reserved.

**Path B: Batch Financial Posting (End-of-Day)**
1. BNPL Transaction Management creates financial posting requests.
2. Integration Middleware writes to `INT_STG.STG_TXN_HEADER` / `STG_TXN_DETAIL`.
3. Control table updated to `READY_FOR_PICKUP`.
4. **CBS EOD batch** (scheduled 22:00 ICT) picks up, validates, and posts to live GL.
5. Status updated to `POSTED`. Merchant sees funds T+1.

## 4. Real-Time Authorization Layer

### 4.1 Pre-Approved Credit Limits
- **Source**: Bank's internal data (salary accounts, deposit history, transaction behavior, existing loan performance).
- **Storage**: Redis (sub-50ms lookup).
- **Update frequency**: Daily batch refresh from bank's data warehouse; real-time decrement/increment during auth.
- **Structure**:
  ```json
  {
    "consumer_id": "C-123456",
    "bnpl_limit_lak": 500000,
    "available_limit_lak": 320000,
    "currency": "LAK",
    "limit_expiry": "2026-09-05",
    "risk_tier": "GREEN"
  }
  ```

### 4.2 Authorization API

**Endpoint:** `POST /api/v1/bnpl/auth`

**Request:**
```json
{
  "auth_id": "AUTH-20260705-001",
  "consumer_id": "C-123456",
  "merchant_id": "M-987654",
  "amount_lak": 180000,
  "currency": "LAK",
  "txn_type": "PURCHASE",
  "channel": "LAOQR",
  "device_fingerprint": "fp_a8f2...",
  "gps_location": "17.9757,102.6331"
}
```

**Response (Approved):**
```json
{
  "auth_id": "AUTH-20260705-001",
  "status": "AUTHED",
  "auth_code": "AUTH-ABC123",
  "approved_amount_lak": 180000,
  "remaining_limit_lak": 140000,
  "settlement_date": "2026-07-06",
  "repayment_date": "2026-08-04",
  "mdr_rate": 0.045,
  "merchant_settlement_lak": 171900,
  "timestamp": "2026-07-05T14:23:11.000Z"
}
```

**Response (Declined):**
```json
{
  "auth_id": "AUTH-20260705-001",
  "status": "DECLINED",
  "reason_code": "INSUFFICIENT_LIMIT",
  "message": "Available BNPL limit (140,000 LAK) is less than transaction amount (180,000 LAK)."
}
```

### 4.3 Authorization State Machine

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ INITIATED│───▶│ PENDING  │───▶│ AUTHED   │───▶│ CONFIRMED│───▶│ SETTLED  │
└──────────┘    └──────────┘    └────┬─────┘    └──────────┘    └──────────┘
                                     │
                                     ├──────▶ CANCELLED (before confirmation)
                                     │
                                     └──────▶ FAILED (fraud, merchant block)
```

| State | Description | Timeout |
|-------|-------------|---------|
| **INITIATED** | Customer selected BNPL at checkout | 5 minutes |
| **PENDING** | Auth request being validated | 10 seconds |
| **AUTHED** | Limit reserved; awaiting merchant confirmation | 30 minutes |
| **CONFIRMED** | Merchant/customer confirmed transaction | Until EOD |
| **SETTLED** | Financial posting completed via EOD | Permanent |
| **CANCELLED** | Auth reversed; limit released | — |
| **FAILED** | Declined or error | — |

## 5. Core Banking Integration: Staging Tables & EOD

This section defines the canonical interface between the BNPL product layer and the bank's CBS. It is based on **Option 1: Staging Table Posting with CBS End-of-Day (EOD) Pickup**.

### 5.1 Principles
1. **Staging tables are the system of record until EOD confirms.**
2. **Never write directly to live GL tables.**
3. **Every staged record must be traceable to a unique BNPL correlation ID.**
4. **EOD pickup is pull-based; the BNPL layer must not push into live ledgers.**
5. **Idempotency is mandatory at every handoff point.**

### 5.2 Staging Table Schema

All staging tables reside in schema `INT_STG` with restricted access.

### 5.3 BNPL-to-CBS Data Flow

#### 5.3.1 BNPL Middleware Write Flow (6 Steps)

**Step 1: Receive & Parse**
- BNPL Transaction Management triggers financial posting (e.g., merchant settlement, customer repayment).
- Payload validated against JSON schema.

**Step 2: Validate Business Rules**
- Account existence check (query read-only account master view).
- Currency validation (LAK only for Phase 1).
- Amount precision (max 4 decimal places).
- Transaction code validity against CBS txn code master.
- Date validation: `SOURCE_TIMESTAMP` must not be future-dated by >24h or back-dated by >90 days.

**Step 3: Enrich**
- Map `TXN_CATEGORY` to GL accounts via BNPL-CBS mapping table.
- Populate `DEBIT_ACCOUNT_BRANCH` / `CREDIT_ACCOUNT_BRANCH` from account master.
- Generate `CORRELATION_ID` = `UUIDv4`.
- Assign `BATCH_ID` = `BNPL_YYYYMMDD_HH` (hourly batches).

**Step 4: Deduplicate**
- Query `STG_TXN_HEADER` for existing `SOURCE_REF_NO`.
- If found with status `POSTED` or `PICKED`: return existing `CORRELATION_ID`.
- If found with status `PENDING` or `HELD`: update if idempotent retry; else return `409 Conflict`.
- If found with status `FAILED` and `RETRY_COUNT < MAX_RETRIES`: allow retry (see Section 5.5).

**Step 5: Write to Staging**
- Open database transaction.
- Insert into `STG_TXN_HEADER` with BNPL extension fields.
- Insert 1..N rows into `STG_TXN_DETAIL` (e.g., principal + MDR split).
- If control record for `BATCH_ID` does not exist: insert into `STG_TXN_CONTROL` with `STATUS = 'OPEN'`.
- Commit transaction.

**Step 6: Finalize Control**
- After all records in logical batch are written, update `STG_TXN_CONTROL`:
  - `STATUS = 'READY_FOR_PICKUP'`
  - `EXPECTED_RECORD_COUNT = COUNT(*)`
  - `EXPECTED_TOTAL_AMOUNT = SUM(TXN_AMOUNT)`
- This signals to EOD that the batch is complete.

#### 5.3.2 EOD Batch Flow (7 Steps)

**Step 1: EOD Initiation**
- Triggered by CBS scheduler at 22:00 ICT.
- Generate `EOD_BATCH_RUN_ID` = `EOD_YYYYMMDD_HHMMSS_<RANDOM>`.

**Step 2: Identify Ready Batches**
```sql
SELECT BATCH_ID 
FROM INT_STG.STG_TXN_CONTROL 
WHERE CONTROL_STATUS = 'READY_FOR_PICKUP'
  AND CREATED_TIMESTAMP < CURRENT_TIMESTAMP - INTERVAL '15' MINUTE;
```
- 15-minute grace period prevents picking a batch still being written.

**Step 3: Lock & Pick Records**
- Update `STG_TXN_CONTROL` to `IN_PROGRESS`.
- Update `STG_TXN_HEADER` for all `PENDING` records in batch to `PICKED`.
- Use `SELECT FOR UPDATE` to prevent race conditions.

**Step 4: Validate Business Rules (Second Pass)**
- Re-validate accounts (closure, dormancy).
- Check available balance for debit transactions.
- Verify no duplicate `CORRELATION_ID` in live `TXN_HISTORY`.
- Check holiday calendar.
- Validate FX rates (if applicable in future phases).

**Step 5: Post to Live GL**
- For each validated record:
  - Insert into live `TXN_HISTORY` / `GL_ENTRIES` via CBS internal APIs.
  - Update account balances via CBS balance update routine.
  - If posting fails, mark individual record as `FAILED` and continue (fail-one-continue-others mode).
  - Alternative: `ALL_OR_NONE` mode configurable per batch.

**Step 6: Update Staging Status**
- Successful: `STG_STATUS = 'POSTED'`, `EOD_POSTING_TIMESTAMP = NOW()`.
- Failed: `STG_STATUS = 'FAILED'`, `EOD_FAILURE_REASON = <error_detail>`.
- Update `STG_TXN_CONTROL` with actuals and `COMPLETED` / `PARTIAL` status.

**Step 7: Reconciliation & Reporting**
- Compare `EXPECTED_*` vs `ACTUAL_*`.
- Insert into `STG_RECONCILE`.
- If mismatch: trigger Critical alert.

### 5.5 Edge Cases & Exception Handling

#### 5.5.1 Duplicate Submission
- **Detection**: Unique constraint on `(SOURCE_SYSTEM, SOURCE_REF_NO)`.
- **Handling**: Return `200 OK` with current status if already posted; `409 Conflict` if logically duplicate.

#### 5.5.2 Retry Policy for Failed Records
- `MAX_RETRIES = 3` (configurable per `TXN_CATEGORY`).
- EOD increments `RETRY_COUNT` on failure.
- If `RETRY_COUNT < MAX_RETRIES`: reset to `PENDING` for next EOD run (requires ops approval or automated retry job).
- If `RETRY_COUNT >= MAX_RETRIES`: move to manual exception queue.

#### 5.5.3 Partial Batch Failure
- **Default mode**: `FAIL_ONE_CONTINUE` (individual failure does not block others).
- `STG_TXN_CONTROL.CONTROL_STATUS = 'PARTIAL'`.
- Failed records remain `FAILED`; operations must resolve before `COMPLETED`.

#### 5.5.4 EOD Batch Run Failure (Crashes Mid-Run)
- **Detection**: `EOD_END_TIMESTAMP` is NULL > 4 hours after `EOD_START_TIMESTAMP`.
- **Recovery**:
  1. Do NOT auto-restart. Operations must investigate.
  2. Identify `PICKED` records for crashed run.
  3. Check live GL for actual postings via `CORRELATION_ID`.
  4. Reset unposted `PICKED` records to `PENDING` (manual approval).
  5. Reset control to `READY_FOR_PICKUP`.
  6. Re-run EOD. Idempotency prevents double-posting.

#### 5.5.5 Reversal of Already-Posted Transaction
- BNPL cannot delete staging records.
- Create **new staging record** with:
  - `TXN_TYPE = 'REVERSAL'` or offsetting debit/credit swap.
  - `ORIGINAL_CORRELATION_ID` in `FLEX_FIELD_4`.
- EOD processes reversal in next run.

#### 5.5.6 Account Closure Between Staging and EOD
- EOD re-validates account status.
- If account closed: mark `FAILED`, reason `ACCOUNT_CLOSED`.
- If account dormant: default to `HELD` pending ops review.

#### 5.5.7 Network Partition Between BNPL and CBS DB
- BNPL middleware queues transactions in local persistent store (Kafka/DB).
- Retry with exponential backoff: 1s, 2s, 4s, 8s, 16s, 32s, then 60s intervals.
- After 10 minutes: circuit breaker opens; return `503` to BNPL services.
- Once restored: drain local queue to staging tables.

#### 5.5.8 Concurrent EOD Instances (Race Condition)
- Implement distributed lock using database advisory locks.
- Lock key: constant `12345` for EOD job.
- Second instance waits 5 minutes; if lock not acquired, exits with error.

#### 5.5.9 Zero-Amount Transactions
- Integration layer rejects `TXN_AMOUNT = 0.0000` unless `TXN_CATEGORY` explicitly allows it.
- Returns `400 Bad Request` with `ERR-ZERO-001`.

#### 5.5.10 CBS EOD Delayed / Missed
- Staging records remain `PENDING`.
- Alert if oldest `PENDING` record > 36 hours.
- Operations can trigger **ad-hoc EOD** for specific batches if CBS supports it.
- Next EOD run processes all accumulated batches.

### 13.4 Operational Runbooks

**Runbook: EOD Job Stuck**
1. Check CBS batch monitor for EOD process status.
2. If running > 6 hours: check DB for blocking locks.
3. If blocked by long-running query: identify session; kill if safe.
4. If EOD process dead: follow Section 5.5.4 (EOD Crash Recovery).
5. Notify CBS team and BNPL integration architect.

**Runbook: High Failed Record Count**
1. Query `STG_ERROR_LOG` for `ERROR_PHASE = 'EOD_POSTING'` in last 24h.
2. Group by `ERROR_CODE` to identify pattern.
3. If `ACCOUNT_CLOSED`: check for bulk account closure event.
4. If `INSUFFICIENT_BALANCE`: check if auto-debit was not reserved.
5. If `FX_RATE_TOLERANCE`: notify treasury.
6. For systemic issues: halt BNPL transaction creation until root cause resolved.

**Runbook: Reconciliation Mismatch**
1. Identify `BATCH_ID` with `RECONCILE_STATUS = 'UNBALANCED'`.
2. Compare `STG_RECONCILE` vs `STG_TXN_CONTROL` totals.
3. If control totals match but live GL differs: investigate CBS double-posting.
4. If staging totals differ from control: investigate BNPL duplicate write.
5. Generate detailed diff report.
6. Escalate to CBS DBA and BNPL integration lead.

**Runbook: Security Incident (Unauthorized Staging Access)**
1. Immediately revoke compromised service account.
2. Query `STG_AUDIT_TRAIL` for all operations by compromised account in last 24h.
3. Identify any unauthorized inserts/updates.
4. If unauthorized records exist: quarantine by setting `STG_STATUS = 'HELD'`.
5. Notify InfoSec, Compliance, and BNPL teams.
6. Force password rotation for all staging schema accounts.
7. Conduct forensic analysis; file incident report.

### Appendix B: BNPL Authorization State Machine

| From State | To State | Trigger | Timeout |
|------------|----------|---------|---------|
| INITIATED | PENDING | Checkout request | 5 min |
| PENDING | AUTHED | Risk check passed | 10 sec |
| PENDING | FAILED | Risk check failed | — |
| AUTHED | CONFIRMED | Merchant/customer confirms | 30 min |
| AUTHED | CANCELLED | Timeout or manual cancel | 30 min |
| CONFIRMED | SETTLED | EOD batch posts | 24h |
| CONFIRMED | FAILED | EOD validation failure | — |
| SETTLED | REFUNDED | Merchant refund request | 30 days |

### Appendix C: EOD Stored Procedure (Pseudocode)

```sql
PROCEDURE BNPL_EOD_PICKUP_AND_POST(
    IN p_eod_batch_run_id VARCHAR(64)
)
BEGIN
    DECLARE v_batch_id VARCHAR(64);
    DECLARE v_cursor CURSOR FOR
        SELECT BATCH_ID 
        FROM INT_STG.STG_TXN_CONTROL 
        WHERE CONTROL_STATUS = 'READY_FOR_PICKUP'
          AND CREATED_TIMESTAMP < CURRENT_TIMESTAMP - INTERVAL '15' MINUTE
        FOR UPDATE SKIP LOCKED;

    OPEN v_cursor;
    FETCH_LOOP: LOOP
        FETCH v_cursor INTO v_batch_id;
        IF done THEN LEAVE FETCH_LOOP; END IF;

        -- Lock control
        UPDATE INT_STG.STG_TXN_CONTROL 
        SET CONTROL_STATUS = 'IN_PROGRESS',
            EOD_BATCH_RUN_ID = p_eod_batch_run_id,
            EOD_START_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id;

        -- Pick records
        UPDATE INT_STG.STG_TXN_HEADER
        SET STG_STATUS = 'PICKED',
            EOD_BATCH_RUN_ID = p_eod_batch_run_id,
            EOD_PICKUP_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id 
          AND STG_STATUS = 'PENDING';

        -- Process each record
        FOR EACH RECORD IN (
            SELECT * FROM INT_STG.STG_TXN_HEADER 
            WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'PICKED'
        ) LOOP
            BEGIN
                -- Validate
                IF NOT validate_account(RECORD.DEBIT_ACCOUNT_NO) THEN
                    RAISE validation_error;
                END IF;
                IF NOT validate_account(RECORD.CREDIT_ACCOUNT_NO) THEN
                    RAISE validation_error;
                END IF;

                -- BNPL-specific validation
                IF RECORD.BNPL_TXN_CATEGORY = 'SETTLEMENT' THEN
                    IF NOT validate_merchant_status(RECORD.BNPL_MERCHANT_ID) THEN
                        RAISE validation_error;
                    END IF;
                END IF;

                -- Post to live GL via CBS API
                CBS_POST_TRANSACTION(RECORD);

                -- Mark posted
                UPDATE INT_STG.STG_TXN_HEADER
                SET STG_STATUS = 'POSTED',
                    EOD_POSTING_TIMESTAMP = NOW()
                WHERE STG_HEADER_ID = RECORD.STG_HEADER_ID;

            EXCEPTION WHEN OTHERS THEN
                UPDATE INT_STG.STG_TXN_HEADER
                SET STG_STATUS = 'FAILED',
                    EOD_FAILURE_REASON = SQLERRM,
                    RETRY_COUNT = RETRY_COUNT + 1
                WHERE STG_HEADER_ID = RECORD.STG_HEADER_ID;

                INSERT INTO INT_STG.STG_ERROR_LOG (
                    STG_HEADER_ID, BATCH_ID, ERROR_PHASE, ERROR_CODE, 
                    ERROR_SEVERITY, ERROR_MESSAGE, ERROR_TIMESTAMP
                ) VALUES (
                    RECORD.STG_HEADER_ID, v_batch_id, 'EOD_POSTING',
                    'EOD-POST-001', 'ERROR', SQLERRM, NOW()
                );
            END;
        END LOOP;

        -- Finalize control
        UPDATE INT_STG.STG_TXN_CONTROL
        SET CONTROL_STATUS = CASE 
                WHEN EXISTS (SELECT 1 FROM INT_STG.STG_TXN_HEADER 
                             WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'FAILED')
                THEN 'PARTIAL' ELSE 'COMPLETED' END,
            ACTUAL_RECORD_COUNT = (SELECT COUNT(*) FROM INT_STG.STG_TXN_HEADER 
                                   WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'POSTED'),
            ACTUAL_TOTAL_AMOUNT = (SELECT SUM(TXN_AMOUNT) FROM INT_STG.STG_TXN_HEADER 
                                   WHERE BATCH_ID = v_batch_id AND STG_STATUS = 'POSTED'),
            EOD_END_TIMESTAMP = NOW()
        WHERE BATCH_ID = v_batch_id;

    END LOOP;
    CLOSE v_cursor;
END;
```
